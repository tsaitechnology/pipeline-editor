01

Install the public packages

The main editor is `@tsai-pe/board`. Add the UI kit, design tokens, data contracts and node registry because host applications provide their own catalog, store and backend implementation.

02

Mount the editor

`pe-board` receives a serializable `Pipeline`. Runtime capabilities are injected: catalog for palette/forms/ports, store for save/open and backend for run/observe/stop. Omit store or backend to hide those controls in read-only or design-only contexts. 

{{ boardSnippet }}
Pass `[pipeline]` to load a document and `[readonly]="true"` to disable editing. Provide `PIPELINE_NODE_CATALOG`, `PIPELINE_STORE` and `PIPELINE_BACKEND` at the host component or route level.
03

Describe your node catalog

The editor does not hard-code integrations. Each node type declares its params, ports and output example. Those fields drive palette groups, inspector forms, dynamic switch outputs and expression help. 

{{ catalogSnippet }}

Params

Use `text`, `number`, `select`, `expression`, `json`, `array`, `secret`, `credential`, `code` and resource picker fields.

Ports

Static, conditional and dynamic ports let nodes model branches, switch cases, split/merge flows and effect-only leaves.

Context

`outputExample` powers `$json`, `$trigger` and `$node["Title"]` suggestions in expression-capable inputs.

04

Connect execution and persistence

The frontend edits graphs and observes runs. Your system owns actual semantics: queueing, credentials, retries, logs and side effects. The `PipelineBackend` contract is intentionally small so it can wrap REST, WebSocket, SSE or local prototypes.

Release and publish

Build through Nx so package dependencies, Angular package outputs and docs stay consistent. The Pages workflow publishes the playground at the repository root and this documentation under `/docs/`.
