01

Install the public packages

The main editor is `@tsai-pe/board`. Add the UI kit, design tokens, data contracts and node registry because host applications provide their own catalog, store and backend implementation.

02

Mount the editor

`pe-board` receives a serializable `Pipeline`. Runtime capabilities are injected: catalog for palette/forms/ports, store for save/open and backend for run/observe/stop. Omit store or backend to hide those controls in read-only or design-only contexts.

Pass `[pipeline]` to load a document and `[readonly]="true"` to disable editing. Provide `PIPELINE_NODE_CATALOG`, `PIPELINE_STORE` and `PIPELINE_BACKEND` at the host component or route level.
03

Understand the integration model

Pipeline Editor is not a hosted automation service. It is a UI kit for product teams that want to embed a workflow builder while keeping execution, credentials, permissions and persistence in their own system.

04

Store pipelines as plain data

A pipeline is a serializable graph: nodes, edges and domain-specific `data`. Keep IDs stable, store snapshots as JSON and run exactly the graph the user saved or submitted.

05

Describe your node catalog

The editor does not hard-code integrations. Each node type declares its params, ports and output example. Those fields drive palette groups, inspector forms, dynamic switch outputs and expression help.

Params

Use `text`, `number`, `select`, `expression`, `json`, `array`, `secret`, `credential`, `code` and resource picker fields.

Ports

Static, conditional and dynamic ports let nodes model branches, switch cases, split/merge flows and effect-only leaves.

Context

`outputExample` powers `$json`, `$trigger` and `$node["Title"]` suggestions in expression-capable inputs.

06

Build inspector forms from params

`ParamField` is the schema for the right-side inspector. Use sections to group advanced settings, `visibleWhen` to keep forms short and object/array fields for structured configuration without writing a custom inspector.

07

Model custom ports and routing

Nodes can expose fixed ports, generate outputs from user-entered array data and reveal conditional outputs when a setting is enabled. This is enough for if/switch nodes, fan-out, split/merge workflows, optional fallbacks and product-specific routing rules.

Static

Use explicit `in`, `out`, `true`, `false` or domain-specific port ids when the shape of the node is known in advance.

Dynamic

Generate one output per item in an array field, such as route cases, departments, locales or model candidates.

Conditional

Reveal ports only when config requires them, keeping simple nodes compact while still supporting advanced branches.

08

Theme and style the editor

Styling is token-driven. Import `@tsai-pe/theme`, scan the shipped Tailwind classes and override CSS variables in your product shell. Board, nodes, forms and UI kit controls read the same tokens.

Dark is the default. Add `.light` to the document root or an app shell to switch token values without replacing components.
09

Expose expressions and variables

Expression fields can be used inside your own inspectors and side panels. Provide a scope from trigger payloads, current `$json` data and upstream node outputs so users get autocomplete for valid references.

$json.customer.email $trigger.body.email $node["Score Lead"].segment
10

Connect execution and persistence

The frontend edits graphs and observes runs. Your system owns actual semantics: queueing, credentials, retries, logs and side effects. The `PipelineBackend` contract is intentionally small so it can wrap REST, WebSocket, SSE or local prototypes.

{{ search() }} $json $trigger $node
11

Keep credentials and permissions server-owned

The editor can render credential selectors, but secret material should never be serialized into `Pipeline.data`. Store references in the pipeline and resolve them inside the backend runtime with the current user's permissions.

Treat every submitted pipeline as user input. Revalidate graph shape, node permissions, credential access and expression safety before starting a run.
12

Start from product recipes

Good host apps usually ship a small catalog of domain recipes instead of exposing generic building blocks first. Model the common flows your users already understand, then let advanced users compose lower-level transform, routing and effect nodes.

Customer ops

Webhook trigger, enrichment, LLM classifier, CRM update, Slack or email notification.

AI agents

Prompt, tool call, branch on confidence, human approval, memory write and final response.

Data pipelines

Fetch, normalize, split rows, transform each item, merge results and write to a warehouse.

Approval flows

Trigger, policy check, conditional route, reviewer task, timeout fallback and final effect.

13

Know the main extension points

The public surface is intentionally small. Most custom behavior comes from catalog metadata, injected backend/store adapters and shared tokens, not from subclassing visual components.

14

Use the UI kit around the editor

The package includes the same controls used by the board: buttons, fields, selects, menus, dialogs, tabs, alerts, badges, tables and layout primitives. They share the same tokens as the canvas.

15

Troubleshoot integration issues

Most integration failures are host-app wiring problems: missing height, missing Tailwind source scanning or intentionally omitted injection tokens.

16

Ship with a production checklist

A strong integration is not just a working canvas. Treat catalogs, migrations, runtime isolation, permissions and observability as part of the product contract.
